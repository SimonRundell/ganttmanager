================================================================================
  1. EXECUTIVE SUMMARY
================================================================================

GanttManager is a web-based project management application designed to
introduce BTec Level 3 Extended Diploma in IT and T-Level Digital Software
Development students to the concepts of Work Breakdown Structure (WBS) and
Gantt Chart tracking. Students create projects, define tasks with
dependencies, assign resources, set milestones, and track progress — observing
in real time how changes to one date cascade through dependent tasks and the
overall timeline.

The system is built on a React + Vite frontend communicating with a PHP/MySQL
REST API backend served by Laragon in development. All authentication is
JWT-based with email verification. Reports are exported as landscape PDFs. The
project is published on GitHub under the Creative Commons
Attribution-NonCommercial-ShareAlike 4.0 licence (CC BY-NC-SA 4.0).


1.1 Scope Boundaries
--------------------------------------------------------------------------------

  * In scope: Full project CRUD, Gantt chart visualisation, dependency management, critical path highlighting, resource allocation, milestone tracking, PDF reports, user authentication, email notifications, teacher/admin oversight dashboard.
  * Out of scope (v1): Real-time collaborative editing (WebSockets), billing, SSO, mobile native apps.
  * Assumed: Students already understand basic project concepts; the app teaches the tooling.

================================================================================
  5. DATABASE SCHEMA
================================================================================

All tables use InnoDB with UTF8MB4 charset. Timestamps default to
CURRENT_TIMESTAMP. Foreign keys use ON DELETE CASCADE unless noted.


5.1 Entity Relationship Overview
--------------------------------------------------------------------------------

  * One User can own many Projects.
  * Projects have many Members (users) via project_members.
  * Each Project has many Tasks, Milestones, Resources and Dependencies.
  * Tasks have a self-referential parent_task_id for WBS hierarchy (summary tasks).
  * Dependencies link a predecessor task to a successor task with a type.
  * Resources are allocated to Tasks via task_resources (with allocation percentage).

================================================================================
  6. BACKEND API SPECIFICATION
================================================================================


6.1 Conventions
--------------------------------------------------------------------------------

  * Every endpoint file begins with:  <?php  require_once '../setup.php';
  * All responses use send_response($data, $code) from setup.php.
  * Success = HTTP 200/201. Client error = 400/401/403/404. Server error = 500.
  * Response envelope: { "status_code": 200, "data": {...}, "message": "ok" }
  * All POST bodies are JSON (Content-Type: application/json).
  * Auth: Bearer JWT in Authorization header (enforced by setup.php enforce_auth_if_needed).
  * authToken.php is the only endpoint exempt from auth (see setup.php allowlist).
  * $receivedData (from setup.php) is used for POST body access; $_GET for query params.
  * Always call log_info() at the start of each endpoint with the received data.
  * Always verify that $GLOBALS["authenticatedUser"] has access to the requested resource.


6.2 Authentication Endpoints
--------------------------------------------------------------------------------

  POST  /auth/register.php
  Purpose  : Register new user. Hash password with password_hash() at cost 12.
             Generate 64-char verify_token via random_bytes(32)->bin2hex(). Send
             verification email via PHPMailer. Return 201 on success.
  Body     : { email, password, screen_name }
  Response : { message: "Verification email sent" }

  POST  /auth/authToken.php
  Purpose  : Login. Validate email/password. Check is_verified=1. Call create_jwt()
             from setup.php. Return token + expiry + user object.
  Body     : { email, password }
  Response : { token, exp, user: { id, email, screen_name, avatar_path, role } }

  GET  /auth/verifyEmail.php?token=XXX
  Purpose  : Look up verify_token. Check expiry. Set is_verified=1. Clear token
             fields. Redirect to {appUrl}/login?verified=1.
  Body     : (query param: token)
  Response : HTTP 302 redirect to frontend

  POST  /auth/forgotPassword.php
  Purpose  : Generate reset_token + reset_token_exp (1 hour). Send reset email
             linking to {appUrl}/reset-password?token=XXX.
  Body     : { email }
  Response : { message: "Reset email sent if account exists" }

  POST  /auth/resetPassword.php

================================================================================
  7. AUTHENTICATION & USER MANAGEMENT
================================================================================


7.1 JWT Flow
--------------------------------------------------------------------------------

  * On login, authToken.php returns { token, exp }.
  * Frontend stores token in localStorage under key: ganttmanager_jwt
  * authStore.js (Zustand) holds { token, user, isAuthenticated }.
  * client.js injects "Authorization: Bearer {token}" on every API call.
  * On 401 response, client.js clears the store and redirects to /login.
  * Token auto-refresh: if token expires within 30 minutes, App.jsx calls refreshToken.php silently on each route change.
  * Teachers/admins are redirected from /dashboard to /teacher immediately after login.


7.2 Registration Flow
--------------------------------------------------------------------------------

  1. User submits email + password + screen_name.
  2. Backend validates: email format, password min 8 chars with at least 1 number and 1 uppercase, unique email.
  3. Backend generates random 64-char hex verify_token, stores with 24-hour expiry.
  4. PHPMailer sends email with link: {appUrl}/verify-email?token={token}
  5. User clicks link -> GET verifyEmail.php -> is_verified set to 1 -> redirect to /login?verified=1
  6. Frontend shows toast: "Email verified! You can now log in."


7.3 Password Rules  (enforced identically by Zod on frontend and PHP on backend)
--------------------------------------------------------------------------------

  * Minimum 8 characters.
  * At least one uppercase letter.
  * At least one number.
  * Stored as bcrypt hash with cost factor 12.


7.4 Avatar Upload Rules
--------------------------------------------------------------------------------

  * Accepted MIME types: image/gif, image/jpeg, image/png.
  * Maximum size: 10 MB (10,485,760 bytes).
  * Backend validates MIME type from file contents using mime_content_type() — NOT from extension.
  * Stored as uploads/avatars/{user_id}.{ext} (overwrites previous avatar).
  * Old avatar file is deleted from disk when replaced.
  * Frontend previews image before upload using FileReader API and rejects invalid files client-side.


================================================================================
  8. PROJECT MANAGEMENT FEATURES
================================================================================


8.1 Work Breakdown Structure (WBS)
--------------------------------------------------------------------------------

Tasks are hierarchical. A summary task (task_type=summary) acts as a
collapsible group header. Its Gantt bar spans the combined start/end range of
all child tasks. Its progress is the weighted average of children by duration.

  * Drag-and-drop reordering updates wbs_order in batch via reorderTasks.php.
  * Indent/outdent sets or clears parent_task_id.
  * Maximum nesting depth: 3 levels (for pedagogical clarity).
  * Deleting a summary task deletes all children (CASCADE in MySQL).


8.2 Dependency Types
--------------------------------------------------------------------------------

+-------+----------------------+--------------------------------------------------+
| Type  | Full Name            | Description                                      |
+=======+======================+==================================================+
| FS    | Finish-to-Start      | Successor cannot start until predecessor finishe |
+-------+----------------------+--------------------------------------------------+
| FF    | Finish-to-Finish     | Successor cannot finish until predecessor finish |
+-------+----------------------+--------------------------------------------------+
| SS    | Start-to-Start       | Successor cannot start until predecessor starts. |
+-------+----------------------+--------------------------------------------------+
| SF    | Start-to-Finish      | Successor cannot finish until predecessor starts |
+-------+----------------------+--------------------------------------------------+

Lag days (positive integer) adds a waiting period after the dependency
condition is met. Negative lag (lead time) allows overlap between tasks.


8.3 Real-Time Dependency Cascade
--------------------------------------------------------------------------------

When a user drags a Gantt bar or edits a task date, the cascade engine runs
client-side first (optimistic UI), then the backend confirms:

  1. Frontend useDependencies.js hook builds a directed graph of all tasks/dependencies in memory.
  2. On date change, a topological traversal propagates new dates to all successors based on dependency type and lag.
  3. Affected tasks are highlighted in amber on the Gantt chart.
  4. A confirmation modal lists all affected tasks: "Changing Task A end date will shift 3 downstream tasks. Apply?"
  5. On confirm, updateTask.php is called with cascade:true.
  6. Backend re-runs the cascade server-side and writes all affected task records atomically.
  7. React Query invalidates the tasks query, triggering a full re-render of the Gantt chart.


8.4 Critical Path
--------------------------------------------------------------------------------

The critical path is the longest chain of dependencies from project start to
end. Any delay to a critical-path task directly delays the project end date.

  * Calculated client-side in useCriticalPath.js using the standard forward/backward pass algorithm.
  * Critical-path tasks are highlighted in red on the Gantt chart (bars and arrows).
  * is_critical=1 is written to the database on each save for audit purposes.
  * Recalculated every time the tasks or dependencies query is refetched.


8.5 Milestones
--------------------------------------------------------------------------------

Milestones are fixed-date markers shown as diamond shapes on the Gantt chart
timeline. They are independent of tasks but visually aligned to the date axis.

  * Achieved milestones are shown in green; overdue milestones in red; future milestones in gold.
  * A milestone may be informally linked to a task by matching dates (not a hard constraint in v1).


8.6 Progress Tracking
--------------------------------------------------------------------------------

  * Each task has a progress field (0-100%).
  * Updated via a slider in the TaskPanel drawer.
  * Summary task progress = duration-weighted average of children (computed on frontend).
  * Overall project progress shown on Dashboard as a donut chart.
  * Tasks set to status=completed automatically set progress=100.


8.7 Resource Management
--------------------------------------------------------------------------------

Resources represent people, equipment or materials assigned to tasks. Each
resource has a colour used to tag Gantt bars and power the resource view.

  * A resource can be assigned to multiple tasks.
  * If a resource's total allocation_pct exceeds 100% for any overlapping date range, show an overallocation warning badge on the resource in the resource view (client-side only in v1).
  * Resource view (tab on GanttPage): horizontal bands per resource showing task bars in resource colour.

  9. GANTT CHART COMPONENT SPECIFICATION
================================================================================


9.1 Layout
--------------------------------------------------------------------------------

GanttChart.jsx is a two-pane layout: a fixed-width left pane (task list) and a
scrollable right pane (timeline canvas). Both panes scroll vertically in sync.
The timeline pane scrolls horizontally independently.

+--------------------------+----------------------------------------------------+
| Property                 | Value                                              |
+==========================+====================================================+
| Left pane width          | 320px fixed                                        |
+--------------------------+----------------------------------------------------+
| Row height               | 40px per task                                      |
+--------------------------+----------------------------------------------------+
| Column width (Day)       | 24px                                               |
+--------------------------+----------------------------------------------------+
| Column width (Week)      | 80px                                               |
+--------------------------+----------------------------------------------------+
| Column width (Month)     | 280px                                              |
+--------------------------+----------------------------------------------------+
| Default zoom             | Week                                               |
+--------------------------+----------------------------------------------------+
| Timeline overrun         | 4 weeks past project end_date                      |
+--------------------------+----------------------------------------------------+


9.2 Gantt Header (Date Ruler)
--------------------------------------------------------------------------------

  * Two rows: top = month name + year; bottom = day numbers or week-starting dates by zoom level.
  * Today is highlighted with a vertical red line across the full chart height.
  * Milestone due dates are shown as gold diamond icons on the date ruler.


9.3 GanttBar (Draggable / Resizable)  — built with react-dnd
--------------------------------------------------------------------------------

  * Bar colour: first assigned resource colour, or project.colour if no resource assigned.
  * Drag left/right: changes start_date and end_date equally (duration preserved).
  * Drag right edge: changes end_date only.
  * Drag left edge: changes start_date only.
  * Summary bars: thicker bar with angled end caps (classic MS Project style).
  * Milestone bars: diamond shape on the date axis (no width).
  * Progress fill: darker overlay inside the bar representing progress %.
  * Critical path bars: red border and slightly darker fill.
  * Tooltip on hover: task title, dates, duration, progress, assigned resources.


9.4 Dependency Arrows (SVG)
--------------------------------------------------------------------------------

GanttDependencyLines.jsx renders SVG arrows on an absolutely positioned layer
spanning the full chart canvas.

  * FS dependency: right edge of predecessor -> left edge of successor (right-angle elbow).
  * Critical path dependencies: red; others: mid-grey (#9CA3AF).
  * Arrowhead markers defined in SVG <defs>.
  * Arrows re-render instantly when tasks are moved (no perceptible lag required).


9.5 Zoom Controls
--------------------------------------------------------------------------------

  * Three levels: Day | Week | Month. Toolbar buttons toggle between them.
  * On zoom change, the chart scrolls to keep "today" visible in the viewport.


9.6 Toolbar Actions
--------------------------------------------------------------------------------

  * + Add Task          : opens TaskForm modal
  * + Add Milestone     : opens milestone form
  * Zoom In / Zoom Out  : cycles through Day / Week / Month
  * Show Critical Path  : toggle red highlights on critical tasks/arrows
  * Export PDF          : calls generateGanttPDF() from pdfUtils.js
  * Filter              : dropdown to filter by status, priority, or resource

13.1 Shared Mailer Factory  (api/mailer.php)
--------------------------------------------------------------------------------

Create api/mailer.php — included by any endpoint needing email. Reads SMTP
config from $config and returns a configured PHPMailer instance.

  ```
  <?php
  function get_mailer($config) {
      $mail = new PHPMailer\PHPMailer\PHPMailer(true);
      $mail->isSMTP();
      $mail->Host       = $config["smtpHost"];
      $mail->SMTPAuth   = true;
      $mail->Username   = $config["smtpUser"];
      $mail->Password   = $config["smtpPassword"];
      $mail->SMTPSecure = $config["smtpSecure"]; // "tls" or "ssl"
      $mail->Port       = $config["smtpPort"];
      $mail->setFrom($config["smtpFromEmail"], $config["smtpFromName"]);
      $mail->isHTML(true);
      return $mail;
  }
  ```

================================================================================
  16. SECURITY CHECKLIST
================================================================================


16.1 Backend
--------------------------------------------------------------------------------

  * All SQL queries use mysqli prepared statements — no string concatenation in queries.
  * Passwords hashed with password_hash($pass, PASSWORD_BCRYPT, ["cost" => 12]).
  * JWT secret: minimum 64-char random string stored only in .config.json.
  * .config.json is in .gitignore and must never be committed.
  * Avatar upload validates MIME via mime_content_type(), not file extension.
  * Avatar filenames are user_id + extension — never user-supplied filenames.
  * Email tokens are random_bytes(32) -> bin2hex() (64 hex chars).
  * Project access: always verify $GLOBALS["authenticatedUser"] has membership before returning project data — except for teacher/admin role which bypasses the membership check.
  * getAllStudents.php and TeacherDashboardPage must return 403 if role=student.


16.2 Frontend
--------------------------------------------------------------------------------

  * JWT stored in localStorage (acceptable for this educational context; document the HttpOnly cookie alternative for any production deployment).
  * All user-supplied text rendered via React JSX (auto-escaped; no dangerouslySetInnerHTML).
  * Zod schemas on frontend mirror PHP validation rules exactly.
  * No API keys or secrets in frontend code or .env files committed to the repository.
  * ProtectedRoute.jsx checks isAuthenticated and role before rendering protected pages.

================================================================================
  18. NOTES FOR THE CODING AGENT
================================================================================


18.1 React / Frontend Standards
--------------------------------------------------------------------------------

  * Functional components only — no class components.
  * All async data fetching via React Query hooks (useQuery, useMutation). Never fetch() directly in a component.
  * All API calls go through functions in src/api/ — these functions use client.js for auth header injection.
  * All date arithmetic goes through dateUtils.js (date-fns wrappers). Never use raw new Date() arithmetic in components.
  * Wrap GanttChart and each page component in an <ErrorBoundary>.
  * Every data-dependent component shows <LoadingSpinner /> while fetching.
  * Every list shows a friendly SVG empty-state illustration when no data exists.


18.2 PHP Standards
--------------------------------------------------------------------------------

  * Every PHP file starts with: <?php  then immediately: require_once '../setup.php'; (adjust path depth).
  * Use $receivedData for POST body access; $_GET for query parameters.
  * Call log_info() at the start of each endpoint with json_encode of received data.
  * Call send_response() for ALL responses — never echo or die() directly.
  * Validate required fields exist before touching the database.
  * For teacher/admin-only endpoints: check $GLOBALS["authenticatedRole"] and return send_response("Forbidden", 403) if role is student.


18.3 Critical Gantt Constraints
--------------------------------------------------------------------------------

  * Handle projects up to 2 years without performance issues — virtualise rows if task count exceeds 100.
  * Dependency arrows must re-render instantly when tasks are moved.
  * Date calculations use calendar days by default (weekends counted). Working-days-only mode is out of scope for v1.
  * Milestone diamonds must always render on top of task bars (highest z-index in the chart layer).
  * Gantt bar drag must snap to whole-day grid positions (no sub-day precision in v1).


18.4 Test Scenarios  (verify these before marking a phase complete)
--------------------------------------------------------------------------------

  * Cascade: create A -> B -> C chain. Move A end date forward 3 days. Confirm B and C shift by 3 days.
  * Circular dependency: attempt to create A->B then B->A. Expect HTTP 400 from createDependency.php.
  * Avatar upload: attempt file > 10 MB — expect client-side rejection before any fetch is made.
  * Avatar MIME: rename a .exe to .jpg and attempt upload — expect server-side rejection via mime_content_type().
  * JWT expiry: set jwtTtlMinutes=1, wait 61 seconds, make any API call — expect 401 and redirect to /login.
  * Teacher access: log in as student, navigate to /teacher — expect redirect to /dashboard.
  * Student isolation: log in as student, manually request getProjects.php?owner_id={other_student_id} — expect only own projects returned.
  * PDF: generate report in Chrome and Firefox. Verify A4 landscape orientation and Gantt chart renders on final page.
